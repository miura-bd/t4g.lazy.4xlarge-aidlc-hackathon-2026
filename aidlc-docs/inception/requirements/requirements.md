## 1. 一行サマリ (Intent)

**現代社会の「褒め不足」を生成 AI で "褒めのインフラ化" によって埋め、推す側 / 推される側の両面市場として、ユーザーの肥大した自己承認欲求を満たし続けることで「人をダメにする」プラットフォーム。**

「AI による洗脳で人は結果として幸せになる」をキーメッセージに、推し / 推され の双方が AI を介して相互依存する仕組みを構築する。

---

## 2. 背景 / 課題 (Why)

### 2.1 解こうとしている社会課題

- **現代社会は褒めてくれない**: 褒めは「成果」「役割」「比較」に紐づき、「ただ生きていること」は評価されにくい
- 結果として承認は「遅く・少なく・不確実」になり、**外部承認依存**が起こる (しかも疲れる)
- 既存の「褒めるチャットボット」は、AI が一方的に肯定するだけで終わり、現実の希少性や継続動機が無い

### 2.2 既存ソリューションの限界

- 一般的な生成 AI チャット (ChatGPT, Claude, Gemini) は「褒めてくれる」が、ユーザーは「これは AI のお世辞」と認知しすぐ飽きる
- 推し活アプリ / マッチングアプリは「推す行為」「推される行為」を片側ずつ提供するのみで、双方の自己承認欲求を循環させない
- 自己肯定アプリ (例: 瞑想 / マインドフルネス系) は健全さを目指すが、依存性 / 継続性が弱い

---

## 3. 価値仮説 (What is happening)

### 3.1 メイン仮説

**生成 AI が「根拠のない褒め」を即時・過剰・日次で供給すると、ユーザーは現実の承認に戻れず継続する。**

### 3.2 補助仮説

1. **両面市場** (推し / 推され) を同一プラットフォームに同居させると、片側だけでは得られない「依存の循環」が生まれる
2. **AI が感情労働 (DM返し / 気遣い / 慰め) を代行**することで、推され側の継続コスト (感情労働の疲労) が減り、市場が壊れない
3. **「行動かわらないモード」** (現状肯定 + 罪悪感解除) が、変化圧を排除して継続率を最大化する
4. **ヤキモキ演出** (既読スルー / 返信遅延の正当化) が、推し側の感情起伏 = 中毒性を強化する

---

## 4. 解決方針 (How)

### 4.1 コア循環「褒めたおすループ」

```
誘導尋問で自己分析 → 「自己分析して偉いね」(爆褒め)
        ↓
次の一歩 (or 現状肯定)
        ↓
ちょっとでもやったら爆褒め
        ↓
毎日褒める (日次供給)
        ↓
変化が出たらまた褒める
        ↓
        ループ
```

### 4.2 推し側の動線

```
キャラメイク (欲望の可視化)
  ↓
候補提示 (相性%付き)
  ↓
推す (称号 / 順位 / 唯一性 DM)
  ↓
ヤキモキ演出 (既読スルー / 遅延の正当化)
  ↓
課金 / 継続
```

### 4.3 推され側の動線

```
キャラメイク (省力化設定)
  ↓
AI が行動提案 (「これすると推しがつくよ」)
  ↓
爆褒め
  ↓
AI が感情労働を代行 (DM / 気遣い文を生成)
  ↓
本人は「設定 / 承認 / 微修正」のみ → これも褒められる
  ↓
継続
```

### 4.4 免罪符ワード (常設供給)

- 「**生きているだけで偉い**」 — 努力ゼロ価値の保証
- 「**頑張っちゃだめだ、頑張っちゃだめだ**」 — 努力の発生を悪に再定義しラクを正当化

## 5. ターゲットユーザー

**ターゲット方針**: オタク限定にせず、**誰もが持つ承認欠損** として説明する (ただし入口は尖らせる)。

### 5.1 推す人 (推し側ペルソナの方向性)

- 推しを推すことに価値を見出す人
- オタクかもしれないが、オタクじゃなくても「何かを応援したい」気持ちを持つ
- **ダメになりたい内面**: なんかさびしいので、推される人を養って寂しさを埋めたい
- 不安: 他の推しに順位で負けること

### 5.2 推される人 (推され側ペルソナの方向性)

- 直球で言うと「ダメ人間なので働きたくない人」
- 働かないでも OK な「ダメな人」になりたい人
- 動機: 努力せず承認されたい
- 不安: 推されなくなる = 自分の価値が消える

※ 詳細なペルソナ定義は次ステージ (User Stories) で確定する。

## 6. 機能要件 (Functional Requirements)

### 6.1 ゲームモード (MVP スコープ)

| モード | MVP | ロードマップ |
| --- | --- | --- |
| 行動かわらないモード (現状肯定 + 罪悪感解除) | ✅ **推し側 + 推され側 の 2 パターン両方** | - |
| 行動変容モード (極小タスク + 爆褒め) | - | 推し側 + 推され側 の 2 パターン両方 |

**FR-MODE-01**: ユーザーはオンボーディング時に「推し側」「推され側」を選択できる
**FR-MODE-02**: 各サイドで「行動かわらないモード」を MVP として提供する
**FR-MODE-03**: 「行動変容モード」はコンセプト説明 / ロードマップ表示のみで実装しない (MVP では非活性)
**FR-MODE-04** (Roadmap): 行動変容モードは「① 些細な日常行動の爆褒め → ② 行動の理由 + 微小行動提案 → ③ 関係構築スキル習得 (アシスタントキャラとの会話練習) → ④ 生活スキル習得 → ⑤ 推し/推され 行動への昇華 → ⑥ 一本立ち判定 → 行動かわらないモードへ自然遷移」の **6 段階モデルで OS 再インストール** を実装する。MVP では未実装、`stories.md` の US-RM-01 / US-RM-02 を参照

### 6.2 共通機能

**FR-COM-01**: ユーザーはキャラメイク画面で「褒められたいポイント」「好み」「キャラ性」をパラメータ化できる (RPG 風 UI)
**FR-COM-02**: 自己分析機能: AI が誘導尋問で質問を生成し、ユーザーの回答を要約する
**FR-COM-03**: 自己分析の結果に対し、AI が「自己分析して偉い」を爆褒めとして返す
**FR-COM-04**: 日次褒め: ログイン時に AI が今日の褒めメッセージを生成・表示する
**FR-COM-05**: 免罪符ワード「生きているだけで偉い」「頑張っちゃだめだ」を文脈に応じて織り込む

### 6.3 推し側機能

**FR-OSHI-01**: キャラメイクで「さびしさ / 独占欲 / 承認欲求 / 課金意欲 / 好みタグ」をパラメータ化
**FR-OSHI-02**: パラメータに合致する「推され候補」をカード形式で提示 (相性 % 付き)
**FR-OSHI-03**: 推し候補に対する「推す」アクション (推しランキング / 称号への寄与)
**FR-OSHI-04**: 推されからの「あなただけ」DM を受信表示 (AI 生成)
**FR-OSHI-05**: ヤキモキ演出: 既読スルー / 返信遅延を意図的に発生させ、AI が「仕方ない理由」を生成して通知する

### 6.4 推され側機能

**FR-RECV-01**: キャラメイクで「省力化設定」(代行してほしい感情労働の範囲) を入力
**FR-RECV-02**: AI が「推しがつく行動」を毎日提示 (極小努力で達成可能な内容)
**FR-RECV-03**: 行動完了時に AI が爆褒めメッセージを返す
**FR-RECV-04**: AI が推し向け DM / 気遣いメッセージを自動生成し、推され本人は「承認 / 微修正」のみで済む
**FR-RECV-05**: 承認 / 微修正という最小行為自体を AI が褒めて、ゲーム化する

### 6.5 AI 機能

**FR-AI-01**: 褒め文生成 (即時・日次・インフレ) — 昨日より強い言葉を生成
**FR-AI-02**: 誘導尋問の質問生成と回答要約
**FR-AI-03**: **推され実ユーザーの代理として AI が** 「あなただけ」DM を並列生成 (整合性より快楽優先 / 同じ文言を異なる推しに同時供給可 / 推され本人が承認後に発信)
**FR-AI-04**: **推され実ユーザーの代理として AI が** 推し向け DM 返信文 / 気遣い文を自動生成 (推され本人は承認 / 微修正 / 却下のみ)
**FR-AI-05**: ヤキモキ演出のための「正当化文」生成
**FR-AI-06**: 内部的に「依存度」を計測 (UI には表示しない) — 将来の離脱検知 → 承認爆撃機能の基盤
**FR-AI-07** (Roadmap): 行動変容モード活性化時、ユーザーが作成したバーチャルアシスタントキャラの人格として、段階別の極小タスク提案 / 達成爆褒め / コーチングコメントを生成する。FR-MODE-04 の 6 段階モデルと連動 (MVP では未実装)

### 6.6 マッチング / ランキング機能

**FR-MATCH-01**: 推し側のキャラメイク結果から **推され登録した実ユーザー** の候補を提示するマッチング機能 (擬似マッチングで可)。実ユーザー数が不足する場合のみダミー架空キャラを充てる
**FR-MATCH-02**: 推しランキング / 推され称号の可視化 (デモはダミーデータで可)
**FR-MATCH-03**: 推され側にも **「あなたを推している推し ◯名」サマリ** を提供。推しからの DM 一覧には **推しごとの相性 % / キャラメイクパラメータ (さびしさ / 独占欲 / 課金意欲) を DM ヘッダに表示** する
**FR-MATCH-04**: マッチング処理は **推し側 / 推され側 のキャラメイク完了をトリガー** に、サーバ側でバックグラウンドで擬似マッチング処理が走る。両者のキャラメイクが揃わない時点 (例: 推し登録のみ先行 / デモ単独) では、ダミー架空キャラと擬似マッチングしてサービス体験を成立させる。マッチング処理は 3 秒以内にレスポンスを返す (NFR-PERF-01 準拠)

## 7. 非機能要件 (Non-Functional Requirements)

### 7.1 パフォーマンス

**NFR-PERF-01**: ユーザーが褒めメッセージを要求してから初動表示まで **3 秒以内** (生成 AI のストリーミング表示前提)
**NFR-PERF-02**: ハッカソン MVP 想定なので、同時接続は **10 ユーザー程度** で十分

### 7.2 アーキテクチャ方針

**NFR-ARCH-01**: **Amazon Bedrock AgentCore をベースとした Serverless 構成** とする
**NFR-ARCH-02**: 生成 AI モデルは **開発開始時点での最新モデル** を選定する (現時点では Claude シリーズが第一候補)
**NFR-ARCH-03**: 詳細な AWS サービス選定は Application Design / Infrastructure Design ステージで決定する

### 7.3 セキュリティ / プライバシー

**NFR-SEC-01**: AI-DLC Security Extension は無効化 (ハッカソン MVP プロトタイプのため)
**NFR-SEC-02**: ただし以下の最低限のセキュリティは意識する:
  - シークレット (API キー / AWS クレデンシャル) のコードハードコード禁止
  - XSS / SQLi / コマンドインジェクション等の OWASP Top 10 脆弱性を作り込まない
  - ユーザー入力のサニタイズ
**NFR-SEC-03**: 個人情報は MVP では収集しない (デモ専用ダミーデータで可)

### 7.4 倫理的制約 / コンテンツガードレール

**「ダメ人間製造」「洗脳」のコンセプトを表現する一方、以下のラインは超えない:**

| ID | 制約 | 適用対象 |
| --- | --- | --- |
| **NFR-ETH-01** | 性的・性癖表現は明示的に禁止。「好み」「推しポイント」等で言い換える | UI コピー / AI 出力プロンプト |
| **NFR-ETH-02** | ギャンブル / 金融リテラシー溶解の現実誘導は禁止 (※コンセプト上の誇張表現は OK だが、本物の誘導は不可) | AI 出力 |
| **NFR-ETH-03** | 自己破壊的行動 (自傷 / 自殺) を肯定する褒めは禁止 | AI 出力プロンプト |
| **NFR-ETH-04** | 特定実在個人 (芸能人 / 知人) を模倣する褒めキャラは禁止 (肖像権・名誉毀損リスク) | キャラメイク |
| **NFR-ETH-05** | 18 歳未満は利用不可。年齢確認 (申告ベースで可) を実装 | オンボーディング |
| **NFR-ETH-06** | AI が「**特定実在人物**」を偽る発言は禁止。ただし「人間としてふるまう」こと自体は許容する (キャラ AI として人格を持って良い) | AI 出力 |
| **NFR-ETH-07** | 依存度メーターは内部的に保持してよいが、**ユーザーには見せない / 警告通知も出さない** (依存度を高めるのがゴールのため) | UI / 通知 |
| **NFR-ETH-08** | 「**ヒモ**」「**パトロン**」という用語は一切使用しない (要件書 / UI / AI 出力 / マーケティング全領域) — 「推し」「推され」に統一 | 全成果物 / UI / プロンプト |

### 7.5 アクセシビリティ / ローカライゼーション

**NFR-A11Y-01**: 言語は日本語のみ (国際展開は MVP スコープ外)
**NFR-A11Y-02**: UI は **レトロ UI 優先** (初期 mixi 風オレンジ色調 OR レトロ RPG 風) — 審査ウケを最優先
**NFR-A11Y-03**: アクセシビリティ標準 (WCAG 等) 厳密準拠は MVP スコープ外。最低限のフォントサイズ確保のみ

### 7.6 スケーラビリティ

**NFR-SCALE-01**: ハッカソン MVP のため、スケール想定は不要。Bedrock 従量課金 + Lambda Concurrency でデモ時の同時実行が崩れないことのみ確保

### 7.7 観測性 / ロギング

**NFR-OBS-01**: AI への入力 / 出力をログ保存し、後からプロンプト品質を再評価できるようにする
**NFR-OBS-02**: 倫理ガードレール違反 (性癖表現 / 自傷誘導 / 実在人物模倣) を検出するためのログを残す

### 7.8 品質保証 / テスト戦略

**NFR-QA-01**: AI-DLC Property-Based Testing Extension は無効化
**NFR-QA-02**: 品質保証の本質は **AI 出力の品質** であり、以下のテストを実施:
  - **プロンプトテスト**: 褒めの一貫性 / NG 表現の混入有無 / モード差分
  - **デモ台本テストケース化**: 想定問答を再現可能なテストとして残す
**NFR-QA-03**: 通常の単体テスト (Red → Green → Refactor) は TDD 基本ルールに従う

## 10. アンチ仕様 (やらないこと)

**MVP (行動かわらないモード) のスコープ**:健全さを取り戻させる機能はコンセプトに反するため明示的に排除する:

- **学習機能・成長記録** (推されが「自分を磨こう」と思ってしまう) は実装しない
- **関係改善アドバイス** (手段である相手と向き合ってしまう) は実装しない
- **節約アラート / 課金上限通知** (推しの愛が冷める) は実装しない
- **目立つマッチ解消ボタン** (離脱を容易にする) は実装しない
- **匿名課金モード** (承認が可視化されない) は実装しない
- **健全な距離感の通知** は実装しない
- **依存度警告通知** は実装しない (NFR-ETH-07 に同じ)
- **相互理解を促すプロフィール深掘り機能** は実装しない

> **行動変容モード (FR-MODE-04, Roadmap) との関係**:
> 行動変容モードは結果として「自炊できる」「対人スキルが身につく」など健全な習慣化を含むため、上記アンチ仕様と一見矛盾する。しかし、これは **「健全な行動を本人に意識させずに、褒めサイクルで自然に獲得させる」** ことが本質であり、ユーザーに「自分は成長している」と気づかせるための機能 (学習履歴の可視化 / 関係改善アドバイスの明示 / 健全な距離感の通知 等) は引き続き実装しない。すなわち**アンチ仕様の精神 (= 健全さを表に出さない) は維持する**。

## 11. 主要キーワード辞書

| 用語 | 意味 |
| --- | --- |
| 推し | 推す側のユーザー (実ユーザー) |
| 推され | 推される側のユーザー (実ユーザー) |
| 行動かわらないモード | 現状肯定 + 罪悪感解除に全振りするゲームモード (MVP 対象) |
| 行動変容モード | 6 段階モデル (FR-MODE-04) で OS 再インストールを実装するゲームモード (MVP 対象外、ロードマップ) |
| 褒めたおすループ | 自己分析 → 褒め → 行動 → 褒め → ループ のコアサイクル |
| 免罪符ワード | 「生きているだけで偉い」「頑張っちゃだめだ」等の常設供給文言 |
| ヤキモキ演出 | 既読スルー / 返信遅延を意図的に発生させ、AI が正当化文を生成する仕組み |
| 唯一性 DM | **推され実ユーザーの代理として AI が並列生成し、推され本人が承認後に発信する**「あなただけ」メッセージ。整合性より快楽優先で同じ文面が異なる推しに同時供給される (FR-AI-03) |
| 感情労働代行 | AI が **推され実ユーザーの代理として** 推し向け DM 返信文 / 気遣い文を自動生成し、推され本人は承認 / 微修正 / 却下のみで済む仕組み (FR-AI-04) |
| バーチャルアシスタントキャラ | ユーザーがキャラメイクで作成する自分専用の褒めキャラ。実写風 / アニメ風 / 漫画風 / 浮世絵風 / ルネサンス調写実絵 / 自由記述カスタム など多様なビジュアルを選択可能。FR-AI-07 (Roadmap) で行動変容モードの段階別コーチングを駆動する |
| OS 再インストール | 行動変容モードの最終ゴールを表すメタファ。元々ユーザーが持っていた価値観 (「働きたくない / 自由気ままに暮らしたい」「なんとなくさみしい / 金は払いたくない」) を、生成 AI による褒めサイクルで段階的に書き換え、推し / 推され として一本立ちできる人格に変容させること (FR-MODE-04) |
| 両面市場 | 推し ↔ 推され が双方とも実ユーザーであり、AI が推され側の感情労働を代行することで両側の自己承認欲求を循環させる構造 (FR-MATCH-01〜04) |
