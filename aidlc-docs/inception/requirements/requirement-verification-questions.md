## Question 4: 生成 AI プロバイダ / モデル

**何を決めるか**: AWS ハッカソンなので **Amazon Bedrock 利用がほぼ必須**。その中でどのモデルを使うか。打ち合わせメモでは TBD。

**5/6 メモ該当箇所** (Section 9 / line 196): "AWS 上で実装"、"ポリーさんに褒めた言葉を喋ってもらう" のみ。モデル選定は **未定**。

A) **Amazon Bedrock + Claude Sonnet** (褒め文・誘導尋問の品質重視)

## Question 7: 倫理的境界線 / コンプライアンス制約

**何を決めるか**: 「ダメ人間製造」「洗脳」を謳う以上、 NG ラインを明示しておく必要がある。打ち合わせでも「性癖は工女両族 NG → 好みに言い換え」が議論された。

**5/6 メモ該当箇所** (line 178): 性癖 → 好み への言い換えが必要、と確定。他は明示されていない。**ハッカソンが社員参加なので「公序良俗」の観点が重要**。

> ※チェックボックス形式。該当するものすべてを `[Answer]:` の後にカンマ区切りで列挙してください。

- [ ] A) **性的・性癖表現は明示的に禁止** (「好み」「推しポイント」等で言い換え)
- [ ] B) **ギャンブル / 金融リテラシー溶解の誘導は禁止** (例:「カジノ行けば」誘導)
- [ ] C) **自己破壊的行動 (自傷 / 自殺) を肯定する褒めは禁止**
- [ ] D) **特定個人 (実在芸能人 / 知人) を模倣する褒めキャラは禁止** (肖像権・名誉毀損リスク)
- [ ] E) **未成年保護**: 18 歳未満は利用不可 / 年齢確認を要件に含める
- [ ] F) **AI が「私は人間です」と偽る発言は禁止** (常に AI であることを明示)
- [ ] G) **依存度を可視化する警告機能を持つ** (「使いすぎ」検知 → 強制ではないが通知)
- [ ] H) **その他** (E で詳細)

[Answer]: A,B,C,D,E 補足）Fは特定の人を偽らなければ、人間としてふるまってよい。Gは内部的にもっているのはいいですが、依存度を高めるのがゴールなので、ユーザー側に伝える必要はありません。

> **AI 補足**: 8 とっぴょうしもないアイデア (Section 8) に「カジノ誘導」が **あえて入っている**点を踏まえると、B は「コンセプト上はそれを誇張表現として扱うが、実装上は本物の誘導はしない」という整理が妥当かもしれません。「ダメ人間を製造する」というコンセプト上、健全さの境界線をどこに引くかはチームで合意必要。

**5/6 メモ該当箇所** (Section 13.5.3): "プロンプト/ポリシーのテスト (褒めの一貫性、NG 表現、モード差分)" と "デモ台本 (想定問答) をテストケース化" が品質保証の方向性。

A) **Yes** — PBT ルールを Blocking 制約として適用 (ビジネスロジック / データ変換 / シリアライゼーション / ステートフルコンポーネントを持つプロジェクト向け推奨)
B) **Partial** — 純粋関数とシリアライゼーションラウンドトリップのみ PBT 適用 (アルゴリズム複雑性が限定的なプロジェクト向け)
C) **No** — PBT ルールをスキップ (シンプルな CRUD アプリ / UI 中心プロジェクト / ビジネスロジックの薄い統合層向け)
E) Other (please describe after [Answer]: tag below)

[Answer]: C (AI 推定 — 要確認) — 本プロジェクトの品質本質は「AI 出力の品質 (褒めの一貫性, NG 表現, モード差分)」であり、これはプロンプトテストで担保すべきもの。PBT は適用範囲が薄い。**ただし「AI に任せる × テストの可視化」が昨年優勝の決め手だった点を踏まえ、ロジック薄くてもテスト戦略は重要**。
