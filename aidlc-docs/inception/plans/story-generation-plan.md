## Q1. ペルソナ数と粒度

**問い**: ペルソナをいくつ作りますか？

- **A**: 2 ペルソナのみ — 推し側代表 1 / 推され側代表 1 (シンプル / 審査向け説明容易)
- **B**: 4 ペルソナ — 推し側 (オタク寄り / 一般寄り) / 推され側 (経験者 / 初心者) の各 2 (尖った入口 + 広い層 をカバー)
- **C**: 3 ペルソナ — 推し側 / 推され側 + ロードマップ向け「行動変容意欲のあるユーザー」(将来拡張を匂わせる)
- **D**: その他 (自由記述)

**AI 下書き**: B (4 ペルソナ)。理由: 要件 5 章「オタク限定にせず誰もが持つ承認欠損として説明する。ただし入口は尖らせる」の二段構えを表現するには、推し側 2 ペルソナ (尖った代表 + 広い層代表) が必要。推され側も「働きたくない人」(直球) と「忙しくて感情労働を外注したい人」(穏当) の 2 ペルソナで両側を見せると審査時の批判耐性が上がる。ただし MVP デモ動線は 2 ペルソナ (各サイドの代表) に絞る。

[Answer]: Aの２ペルソナで。推し側と推され側の２つの例を出したい。

## Q2. ペルソナに含める属性項目

**問い**: 各ペルソナに記載する属性はどこまで含めますか？

- **A**: 必須のみ — 名前 / 年齢 / 職業 / 1 行動機 / 不安要素
- **B**: 標準 — A + 利用シーン / 利用頻度 / 課金可能性 / デバイス / 1 日の生活パターン
- **C**: 詳細 — B + 心理特性 (MBTI 風) / SNS 利用傾向 / 過去の課金履歴想定 / 推し活経験
- **D**: その他 (自由記述)

**AI 下書き**: B (標準)。理由: 審査向けに「物語性」を出しつつ、ハッカソン MVP に必要な最小限の解像度。MBTI 等は審査で揚げ足取りリスクがある一方、利用シーン / 利用頻度 は AI 出力プロンプト設計に直接効くので必須。年齢は NFR-ETH-05 (18 歳未満不可) を満たす範囲で設定。

[Answer]: B ＋　MBTIとSNS利用動向を追加する形で。

## Q3. ストーリーの総数と粒度

**問い**: ユーザーストーリーは何本作りますか？ MVP のみ / ロードマップ含む の方針もここで確定します。

- **A**: 9 ストーリー — 打ち合わせメモ 12.2 の US-01〜US-09 をそのまま採用 (MVP のみ)
- **B**: 12 ストーリー前後 — 共通 3 + 推し側 4 + 推され側 4 + ロードマップ参考 (行動変容モード) 1〜2 (MVP + 拡張示唆)
- **C**: 15 ストーリー以上 — 細分化して INVEST の "Small" を厳守、AI 任せ範囲も独立ストーリー化
- **D**: エピック構造 — 「キャラメイク」「褒めたおすループ」「両面市場」「AI 感情労働代行」をエピックとし、配下に 2〜3 ストーリーずつ
- **E**: その他 (自由記述)

**AI 下書き**: B (12 ストーリー前後)。理由: メモ 12.2 の 9 ストーリーは推し / 推され で分かれているが、共通機能 (キャラメイク / 自己分析 / 日次褒め / 免罪符) が双方に重複しているため、共通ストーリー 3 本に統合し、推し固有 4 / 推され固有 4 / ロードマップ 1〜2 の構成が整理しやすい。エピック構造 (D) は審査用には冗長なため見送り。

[Answer]: E 審査員への読みやすさもあるので、数は減らしたい
共通 2 + 推し側 2(ソフトとハードで１つずつ) + 推され側 2(ソフトとハードで１つずつ)のような形で。
推し(1) かなり経済的に余裕があるので、ズブズブに「推し」の生活を支援して感情を充足されたい (ハード推し)

推し(2) ちょっと経済的に余裕があるので、「推し」の生活をほどほどに支援して毎日の生活の中でちょっとぐらい感情を充足されたい (ソフト推し)

上記のような意味でのソフトとハードで、推され側も考えたい。

## Q4. ストーリーのブレイクダウン方式

# Story Breakdown Approach Comparison (参考)

| 方式 | 章立てイメージ | 利点 | 欠点 | 本案件適合度 |
| --- | --- | --- | --- | --- |
| Persona-Based | 推し側 / 推され側 / 共通 | 両面市場が一目で伝わる、ペルソナと FR が対応 | ジャーニー横断で重複が出やすい | ◎ |
| Journey-Based | オンボーディング → キャラメイク → 〜 | ユーザー体験の連続性、デモ台本に直結 | 推し / 推され どちらの動線か明示しないと混乱 | ○ |
| **Hybrid (Persona + Journey)** | 章: ペルソナ / 章内: ジャーニー順 | 両面市場の構造 + 体験連続性 を両立 | 構造がやや複雑 | **◎ (Q4 推奨)** |
| Feature-Based | FR-COM / FR-OSHI / FR-RECV / FR-AI | 要件書と 1:1 対応、実装分担が容易 | 物語性が消える、審査向けには弱い | △ |
| Epic-Based | 褒めたおすループ / 両面市場 / 感情労働代行 | コンセプト主導、訴求力◎ | 9 ストーリー規模では過剰構造 | △ |

- ペルソナごとに: ID / 名前 (匿名で可) / 年齢 / 職業 / 利用シーン / 動機 / 不安 / 利用頻度 / 課金可能性 / 紐付くストーリー ID 群

- デモ動線対応表 (Q9 で確定)
- 倫理ガードレール対応表 (Q8 で確定)
