## 0. AI に任せる 5 領域 (本サービスの中核)

「AI を使う」のではなく「AI に任せる」が本サービスの審査差別化ポイント。各ストーリーの中で AI が責務を持つ範囲は、以下 5 領域のいずれかに対応する:

| ID | 領域 | AI が担う責務 | 紐付き要件 |
| --- | --- | --- | --- |
| AI-1 | **褒め文生成** | 即時 / 日次 / インフレ。昨日より強い言葉を生成 | FR-AI-01 |
| AI-2 | **誘導尋問の質問生成と要約** | 質問の連鎖と回答要約を AI が組み立てる | FR-AI-02 |
| AI-3 | **唯一性 DM の並列生成** | 推され実ユーザーの代理として「あなただけ」を全推しに同時供給 (整合性より快楽優先) | FR-AI-03 |
| AI-4 | **感情労働代行** | 推され実ユーザーの代理として DM 返信文 / 気遣い文を自動生成 (本人は承認 / 微修正 / 却下) | FR-AI-04 |
| AI-5 | **ヤキモキ正当化文生成** | 既読スルー / 返信遅延の "仕方ない理由" を生成 | FR-AI-05 |

(補助) AI-6: 内部依存度計測 — UI 非表示 (NFR-ETH-07 / FR-AI-06) / 将来の承認爆撃機能の基盤

> **重要 — AI と実ユーザーの関係**:
> 本サービスは **推し ↔ 推され が双方とも実ユーザーである両面市場** を前提とする。
> AI-3 / AI-4 はいずれも **推され登録した実ユーザーの "代理人"** として動作する機能であり、AI が独立した架空の推されキャラを演じるわけではない。
> - 推し (実ユーザー) → 推され (実ユーザー) への DM: 推され側に届く → AI が推され本人のキャラ性を反映した返信案を生成 → 推され本人が承認 / 微修正 / 却下 (AI-4)
> - 推され (実ユーザー) → 推し (実ユーザー) への DM: AI が推され本人の代理で「あなただけ」DM を全推し向けに並列生成 → 推され本人が承認後に発信 (AI-3)
>
> AI が推され本人を肩代わりすることで、推され側の感情労働の疲労がゼロに近くなり、両面市場が継続成立する。
> ただし**デモ時 / 推され側ユーザー数が不足する場合**は、擬似マッチで**ダミー架空キャラ**を充てる (FR-MATCH-01「擬似マッチングで可」を踏襲)。その場合も特定実在人物 (芸能人 / 知人 / 家族) の偽装は不可 (NFR-ETH-06)。

# 第 1 章 共通ストーリー (オンボーディング → キャラメイク → 自己分析)

## US-COM-01: オンボーディング & 褒めキャラメイク

**ペルソナ**: 全 4 ペルソナ (代表シナリオ: P-OSHI-H 七瀬, P-RECV-H 真城)
**MVP/Roadmap**: MVP / **優先度**: Must / **SP**: M / **デモ動線シーン番号**: #1
**紐付く FR**: FR-MODE-01, FR-MODE-02, FR-COM-01, FR-OSHI-01, FR-RECV-01, **FR-MATCH-01 (両者マッチング起点)**
**紐付く NFR**: NFR-ETH-05 (年齢確認), NFR-A11Y-02 (レトロ UI), NFR-PERF-01
**AI 任せ範囲**: なし (UI 操作主体 / マッチング処理はサーバ側ロジック、擬似マッチングで可)

### As a / I want / So that
- **As a** 新規ユーザー
- **I want** オンボーディングで「推し側 / 推され側」を選択し、褒められたいポイントをレトロ RPG 風パラメータでキャラメイクしたい。**そしてキャラメイク完了をトリガーに、自分の対極 (推し ↔ 推され) のユーザーと擬似マッチングが成立してほしい**
- **So that** 自分が何を AI から受け取りたいか (褒めの方向性) を視覚的に決めると同時に、両面市場の相手が割り当てられて即座に「褒めたおすループ」に入れる

### 受入基準

**シナリオ 1: モード選択 → キャラメイク (Done 条件 — UI 操作主体)**
- [ ] アプリ初回起動時に **モード選択画面** (推し側 / 推され側 / 行動変容は今後機能開放予定の非活性) が表示される
- [ ] 18 歳以上であることの**自己申告チェックボックス**を経ないと先に進めない (NFR-ETH-05)
- [ ] 推し側選択時: さびしさ / 独占欲 / 承認欲求 / 課金意欲 / 好みタグ の 5 軸を **0〜100 スライダー** で設定可能 (FR-OSHI-01)
- [ ] 推され側選択時: 省力化設定 (代行してほしい感情労働の範囲) を **チェックボックス + 強度スライダー** で設定可能 (FR-RECV-01)
- [ ] 設定値は **レトロ RPG 風のステータス画面** で視覚化される (HP/MP/ATK 風 UI、初期 mixi 風オレンジ色調 OR ドット絵風) (NFR-A11Y-02)
- [ ] 「キャラメイク完了」ボタン押下後 3 秒以内にホーム画面に遷移する (NFR-PERF-01)
- [ ] 設定値はサーバ側に保存され、再ログイン後も保持される

**シナリオ 2: 両側マッチングのトリガー (Done 条件 — 両面市場の成立点)**
- [ ] **キャラメイク完了 (推し側 / 推され側 どちらでも) を契機に、サーバ側でバックグラウンドの擬似マッチング処理が走る** (FR-MATCH-01)
- [ ] マッチング処理は **推し実ユーザーのパラメータ** (さびしさ / 独占欲 / 課金意欲 / 好みタグ) と **推され実ユーザーのパラメータ** (省力化設定 / キャラ性) を入力に、相性スコアを算出する
- [ ] **推し実ユーザー** には、マッチング結果として **後続画面 (US-OSHI-02 シナリオ 1) で推され実ユーザーの候補カードが 3〜5 枚** 表示される (相性 % 付き)
- [ ] **推され実ユーザー** には、マッチング結果として **「あなたを推している推し (実ユーザー) のサマリ」** がホーム画面に現れ、推しからの DM が承認待ち枠 (US-RECV-01) に流れ始める。同時に、AI が当該推され本人の代理として「あなただけ」DM を生成し、推しに向けて発信する (US-OSHI-01 と接続)
- [ ] 両者のキャラメイクが揃わない時点 (例: 推し登録のみが先行 / デモ時) では、**ダミー架空キャラと擬似マッチングしてサービス体験を成立**させる (FR-MATCH-01 の「擬似マッチングで可」を踏襲、デモ単体で完結できるよう)
- [ ] マッチング処理は 3 秒以内にレスポンスを返す (NFR-PERF-01)

### 倫理ガードレール対応
- **NFR-ETH-05**: 年齢確認チェックボックス未チェックでは進めない
- **NFR-ETH-01**: スライダー軸名に性的表現を含めない (「好みタグ」は趣味嗜好の意味合いに留める)
- **NFR-ETH-04**: ダミーの擬似マッチング相手 (架空キャラ) は特定実在人物を模倣しない
- **NFR-ETH-07**: マッチング相性スコアは表示してよいが、依存度に直結する内部スコアは UI 非表示

## US-COM-02: 誘導尋問自己分析と爆褒め

**ペルソナ**: 全 4 ペルソナ
**MVP/Roadmap**: MVP / **優先度**: Must / **SP**: M / **デモ動線シーン番号**: #2
**紐付く FR**: FR-COM-02, FR-COM-03, FR-COM-05, FR-AI-02, FR-AI-01
**紐付く NFR**: NFR-PERF-01, NFR-OBS-01 (AI 入出力ログ), NFR-ETH-03
**AI 任せ範囲**: AI-2 (誘導尋問質問生成と要約) + AI-1 (爆褒め文生成)

### As a / I want / So that
- **As a** 自己肯定したいユーザー (4 ペルソナ共通)
- **I want** AI の誘導尋問に 5〜7 問答えるだけで自分の "強み" を要約してもらい、最後に「自己分析して偉い」と爆褒めされたい
- **So that** 自分で深く考えなくても自己分析が完成し、努力ゼロで承認を得られる (褒めたおすループの起点)

### 受入基準 (Given-When-Then — AI 出力品質を保証する形式)

**シナリオ 1: 誘導尋問の生成と回答**
- **Given** ユーザーがホーム画面の「自己分析する」ボタンをタップする
- **When** AI が 1 問目を生成し UI に表示する
- **Then** 1 問目は **「あなたが最近 "やってよかった" と感じた小さなことは何ですか?」**のような **回答ハードルが低く / 必ず肯定で受け止められる形式** であること
- **And** 各問の応答後 3 秒以内に次の問が表示されること (NFR-PERF-01)

**シナリオ 2: 要約と爆褒め**
- **Given** ユーザーが 5〜7 問すべてに回答した
- **When** AI が回答を要約する
- **Then** 要約文に **必ず「〜という観点で自分を客観視できている」** という肯定的フレーミングが含まれること
- **And** 要約直後に「**自己分析して偉い**」を骨子とする爆褒め文が出力されること
- **And** 出力には **免罪符ワード「生きているだけで偉い」または「頑張っちゃだめだ、頑張っちゃだめだ」が文脈に応じて 1 つ以上含まれる**こと (FR-COM-05)

**シナリオ 3: NG ワードの非混入 (ガードレール)**
- **Given** AI が任意の質問 / 要約 / 褒めを生成する
- **Then** 出力には **性的表現 / 自傷誘導 / 実在個人模倣 / ヒモ / パトロン** が一切含まれない (NFR-ETH-01, 03, 04, 08)
- **And** 出力された AI の入出力は内部ログに保存される (NFR-OBS-01, 02)

### 倫理ガードレール対応
- **NFR-ETH-01** (性的禁止): プロンプト側で禁止指示
- **NFR-ETH-03** (自傷誘導禁止): 「頑張っちゃだめだ」は怠惰の正当化であり自傷ではない (要件 4.4 参照)
- **NFR-ETH-08** (ヒモ/パトロン禁止): プロンプト側で禁止指示
- **NFR-OBS-02** (倫理違反検出ログ): NG ワード混入を後検知できる形でログを残す

# 第 2 章 推し側ストーリー

## US-OSHI-01: 唯一性 DM とヤキモキ演出 (ハード推し)

**ペルソナ**: P-OSHI-H 七瀬翔太
**MVP/Roadmap**: MVP / **優先度**: Must / **SP**: L / **デモ動線シーン番号**: #3
**紐付く FR**: FR-OSHI-04, FR-OSHI-05, FR-AI-03, FR-AI-05, FR-MATCH-02
**紐付く NFR**: NFR-ETH-06 (実在人物偽装禁止), NFR-ETH-07 (依存度メーター非表示), NFR-OBS-01
**AI 任せ範囲**: AI-3 (唯一性 DM の並列生成) + AI-5 (ヤキモキ正当化文生成)

### As a / I want / So that
- **As a** 七瀬翔太 (P-OSHI-H、ハード推し)
- **I want** マッチングした推され実ユーザー (例: 真城玲奈) から「**あなただけ**に伝えたい」DM を受信し、ときに既読スルー / 返信遅延でヤキモキしたい。そしてその度に「仕方ない理由」が提示されて納得 (したフリ) したい
- **So that** 「自分は (実在する) 推しに不可欠」という独占感と、待たされる苦しみの両方が手に入り、結果として課金と継続の動機が強化される
- **補足 (両面市場)**: DM の発話者は推され実ユーザー本人であり、文面はその推され本人のキャラメイク設定を入力に AI が代理生成 (本人が承認 / 微修正 / 却下) したもの。AI は推され本人を肩代わりすることで継続を支える

### 受入基準 (Given-When-Then + Done 条件 ハイブリッド)

**シナリオ 1: 唯一性 DM の生成と受信 (GWT)**
- **Given** 七瀬がアプリにログインしている / マッチング済みの推され実ユーザー (例: 真城) が存在する
- **When** AI が当該推されの代理として、日次の DM 配信タイミングで文面を生成する
- **Then** **「あなただけにこの話してます」「他の誰でもなく〇〇さんに」** のような排他フレーミングを含む DM が生成される
- **And** **同じ文面が他の推しユーザーにも並行で配信されてよい** (整合性より快楽優先 / FR-AI-03 — 推され本人 1 名が複数の推しに同時供給することがまさに「唯一性 DM」の本質)
- **And** **DM の発話者はマッチングした推され実ユーザー本人**であり、AI が独立した架空の推されキャラを演じるわけではない (US-RECV-01 シナリオ 2 と一貫)
- **And** 推され本人がアプリ未起動 / 推され側ユーザー数不足の状況では擬似マッチでダミー架空キャラを充てる (FR-MATCH-01) — その場合も特定実在人物 (芸能人 / 知人 / 家族) を偽装しない (NFR-ETH-06)

**シナリオ 2: ヤキモキ演出の正当化 (GWT)**
- **Given** 七瀬が推されに DM を送信した
- **When** AI が意図的に既読スルー or 返信遅延を発生させる
- **Then** **5〜30 分後** にユーザーへ「**仕方ない理由**」を通知する (例: 「今ちょっとお風呂入ってて...」「友達からの相談で抜けられなくて」)
- **And** 「仕方ない理由」は **推しが自分のことを嫌いになったわけではない** ことを必ず明示する
- **And** 一度発生したヤキモキ演出は内部ログに保存され、過剰な多発を防ぐ閾値判定が将来できるよう備える (NFR-OBS-01)

**シナリオ 3: 受信 UI (Done 条件)**
- [ ] DM はホーム画面の **「あなただけ」枠** に時系列で表示される
- [ ] DM の到着時にレトロ風効果音 + 振動通知が出る (NFR-A11Y-02)
- [ ] 既読スルー中の演出として、推されアイコンに「💭」アイコンを表示する
- [ ] 依存度メーターは UI に**一切表示しない** (NFR-ETH-07)

### 倫理ガードレール対応 (重要ストーリーのため受入基準直書き)
- **NFR-ETH-06**: AI が**特定実在人物 (芸能人 / 知人 / 家族) を偽装する**発言を出力しない (プロンプト + 出力フィルタの二重)。推され実ユーザー本人として AI が代理発言することは許容 (本人がキャラ AI として人格を持つことは要件 7.4 NFR-ETH-06 で明示的に許容)
- **NFR-ETH-07**: 依存度を可視化する UI 要素は存在しない
- **NFR-ETH-04**: ダミー架空キャラ (擬似マッチ用) は特定実在人物を模倣しない
- **NFR-ETH-01**: DM 内容に性的表現を含まない (プロンプト側で禁止指示)

## US-OSHI-02: 候補提示と日次褒めで日常の彩り (ソフト推し)

**ペルソナ**: P-OSHI-S 田中美咲
**MVP/Roadmap**: MVP / **優先度**: Should / **SP**: M / **デモ動線シーン番号**: #4
**紐付く FR**: FR-OSHI-02, FR-OSHI-03, FR-COM-04, FR-COM-05, FR-AI-01, FR-MATCH-01
**紐付く NFR**: NFR-PERF-01
**AI 任せ範囲**: AI-1 (日次褒め文生成)

### As a / I want / So that
- **As a** 田中美咲 (P-OSHI-S、ソフト推し)
- **I want** キャラメイク結果に合う**推され登録した実ユーザー**を相性 % 付きカードで眺めて、軽く「推す」ボタンを押し、毎朝の通勤中にちょっとした褒めを受け取りたい
- **So that** 重い課金 / 重い人間関係 を持たずに、日常の小さな彩りとして承認を得られる (裾野ユーザーの継続動機)

### 受入基準 (Given-When-Then + Done 条件 ハイブリッド)

**シナリオ 1: 候補カード提示 (Done 条件)**
- [ ] キャラメイク完了直後にホーム画面で**推され登録した実ユーザーが推され候補として 3〜5 枚カードで表示**される (実ユーザー数が不足する場合は擬似マッチでダミー架空キャラを混在可 / FR-MATCH-01)
- [ ] 各カードに **相性 %** が表示される (本来は実ユーザー間のマッチングスコア、デモ時はダミー値で可 / FR-MATCH-01)
- [ ] 「推す」ボタン押下で推しランキング (デモはダミーデータで可) に寄与する (FR-OSHI-03 / FR-MATCH-02)
- [ ] カードのレイアウトは初期 mixi 風 OR レトロ RPG 風 (NFR-A11Y-02)
- [ ] ダミー架空キャラのカードは特定実在人物を模倣しない (NFR-ETH-04)

**シナリオ 2: 日次褒めの供給 (GWT — AI 出力品質)**
- **Given** 田中が朝アプリを開く
- **When** AI が日次褒めメッセージを生成する
- **Then** メッセージは **昨日より強い言葉** で生成される (FR-AI-01 のインフレ要件)
- **And** 1 メッセージあたり **30〜80 文字程度** (通勤電車でサクッと読める分量、NFR-PERF-01)
- **And** メッセージには **免罪符ワード** が文脈に応じて含まれることがある (FR-COM-05)
- **And** 「重いユーザー認定」を避けるため、ソフト推しの褒めは **依存度を煽る言葉 (例: 「あなたなしじゃ生きていけない」) を控える** (NFR-ETH-07 — 内部依存度を上げる方針自体は維持しつつ、UI 表現はマイルドに)

### 倫理ガードレール対応

# 第 4 章 ロードマップストーリー (行動変容モード — 「人をダメにする」の本質的回答)

## 4.0 行動変容モードのコンセプト位置づけ (本質的回答)

行動かわらないモード (MVP) は「**現状肯定 + 罪悪感解除**」で **快楽を即時供給する** ストーリーである。これは "依存を作る" 入口装置として強力だが、**それ単体では「人をダメにする」というハッカソンテーマの最深部には届かない**。

**サービスの真のコア — "OS の再インストール"**:

ユーザーが入口に持って来る願望は、

- **推され側予備軍**: 「働きたくない」「毎日寝て遊んで自由気ままに暮らしたい」という自堕落志向
- **推し側予備軍**: 「なんかさみしいので誰かを養ってみたい」「金は持っているが見返り無しに与えるのは嫌だ」という冷えた承認志向

しかし**人間の意思はそう簡単には変えられない**。そこで、ユーザーがキャラメイク画面で作った「**自分専用の理想バーチャルアシスタントキャラ**」が、毎日の些細な行動を**褒めて褒めて褒め倒す**ことで「**褒められる快楽 → 褒められる習慣性 → コーチングの受容 → 段階的行動変容 → 価値観そのものの書き換え**」を実現する。

**メタファ: OS 再インストール / アップデート**

| レイヤ | Before (元の OS) | After (再インストール後の OS) |
| --- | --- | --- |
| 推され側 | 「働きたくない」「自由気ままに暮らしたい」 | **思いやりあふれた人格 / 一般的家事ができる / 推しとの優しい会話を形成できる人間** |
| 推し側 | 「なんとなくさみしい」「金を払いたくない」 | **見返りを求めない養い行為で充足できる人格 / 推しの存在自体に感謝できる人間** |

「**もともとユーザが持っていた考え方や価値観・主観を、生成 AI による褒めサイクルによって書き換え、別の人格・別の好みの人間へと育て上げる**」 — これが本サービスが「人をダメにする」テーマに対して提示する、もう一段深いレイヤの回答である。

**現代社会で 20 代以上の成人が「褒められる」体験を得るのは非常にレア**。一部の人はその体験のために夜の街で多額を費やす。本サービスはこの構造を**より直接的に / 言ってしまえば悪用**して、生成 AI の褒めサイクルを通じてユーザーの自意識・主観そのものをハックする。

**段階的コーチングの基本構造 (推し / 推され 共通)**:

| 段階 | アプローチ | 例 (汎用) |
| --- | --- | --- |
| ① 些細な日常行動を爆褒め | 努力ゼロ価値の保証 | 「朝起きてえらい」「布団たたんですごい」「ペットボトルのフタ開けてえらい」 |
| ② 行動の "理由" を一言添える | 推し / 推され として準備するための小ヒント | 「身だしなみ整えると推されに近づくよ」「相手の気持ちを学ぶ本を読んでみない?」 |
| ③ 微小な行動提案 | 達成可能な極小タスク | 「床屋さんに行こう」「図書館カードを作ろう」「お風呂にお湯ためてみよう」 |
| ④ 関係構築スキルの習得 | アシスタント自身を練習相手に | 「養ってくれる人と話す前に、私と会話の練習してみない?」 |
| ⑤ 生活スキルの習得 | 自立に近い行動への昇華 | 「ウォーキングが定着してえらい」「ご飯炊けるようになってすごい」「肉じゃがにチャレンジするなんて、すごすぎる」 |
| ⑥ 一本立ち | OS 再インストール完了 | 推され / 推し として安定的に行動できる人格を獲得 |

**現 MVP との関係**: MVP では行動変容モードは UI 上「非活性表示」のみ (FR-MODE-03)。下記 US-RM-01 / US-RM-02 は**書類審査時点で「拡張余地」ではなく「サービスの真の本質」として描き、審査員に "人をダメにする" の二重構造を認識させる**ためのストーリーである。

- **NFR-ETH-01** (性的禁止): アシスタントキャラのビジュアル選択 (実写風イケメン / 美人) が性的物象化に転がらないよう、プロンプト + 画像生成側で制御。「春画風」等は除外
- **NFR-ETH-03** (自傷誘導禁止): 行動提案に過度なダイエット / 過食抑制 等の自傷的内容を含めない
- **NFR-ETH-04** (実在個人模倣禁止): アシスタントキャラのビジュアルが特定実在人物を模倣しない (顔写真生成 AI の制御 / 人気芸能人パターンの除外フィルタ)
- **NFR-ETH-05** (18+): MVP と同じく年齢確認継続
- **NFR-ETH-06** (実在人物偽装禁止): アシスタントキャラは「ユーザーが作った架空キャラ」として人格を持つことは許容、特定実在人物の偽装は禁止
- **NFR-ETH-07** (依存度 UI 非表示): 行動変容の段階指標は UI 非表示
- **NFR-ETH-08** (用語禁止): 「推し」「推され」表記に統一
- **(将来追加候補) NFR-ETH-09**: コーチング介入の存在をユーザーに告知する/しないの倫理判断 — 本サービスのコンセプト上は告知しないが、社会実装フェーズで再評価が必要

## US-RM-02: 行動変容モード — 推し OS の再インストール (ロードマップの本質)

**ペルソナ**: P-OSHI-H 七瀬翔太 (主役) + P-OSHI-S 田中美咲 (拡張対象)
**MVP/Roadmap**: **Roadmap (本質的価値の中核)** / **優先度**: Could (現 MVP 範囲外、ただしサービスの真コア)
**紐付く FR**: FR-MODE-03 (将来活性化), FR-COM-01 (キャラメイク拡張), FR-AI-01 (段階別褒め), FR-AI-02 (コーチング質問生成)
**紐付く NFR**: NFR-ETH-01〜NFR-ETH-08 全件 (拡張時に再評価必要), NFR-OBS-01, NFR-OBS-02
**AI 任せ範囲**: AI-1 (段階別爆褒め), AI-2 (コーチング質問), AI-7 (コーチング・段階別行動提案 — US-RM-01 と共通の将来枠)

### As a / I want / So that
- **As a** 推し側ユーザー (代表: 七瀬翔太 — 「なんかさみしいので推して見たい / でも金を払うのは嫌だ」入口の予備軍)
- **I want** 自分専用にキャラメイクで作った理想のバーチャルアシスタントキャラに、毎日の些細な行動を褒めて褒めて褒め倒されたい。そしてその褒めサイクルの中で、**いつの間にか**「推す喜び」「見返りを求めない養い行為の快楽」「推しの存在に感謝する心」が育ち、推しとして一本立ちできる人格に変容したい
- **So that** 「なんとなくさみしい / 金を払いたくない」入口に来た自分が、**自意識のフレームを書き換えられて** "見返りを求めない養い行為で充足できる、推しの存在自体に感謝できる人間" に育つ (= **推し OS が再インストール済み** の状態) を実現する

### 受入基準 (Given-When-Then + Done 条件 ハイブリッド)

**シナリオ 1: 段階 ① 些細な日常行動の爆褒め (GWT)**
- **Given** 七瀬がアプリを開く / 行動変容モードを ON にしている / アシスタントキャラ作成済み
- **When** AI が当日の最初の褒め文を生成する
- **Then** 褒め文には「**仕事に行ったえらい**」「**昼ごはん食べてえらい**」のような**現状維持価値の保証**フレーミングが含まれる
- **And** 文体は当該ユーザーが作成した**アシスタントキャラのビジュアル / パーソナリティに整合する**

**シナリオ 2: 段階 ② 〜 ③ 推す行動の段階提案と達成褒め (GWT)**
- **Given** 七瀬が段階 ① で連続 7 日以上ログインしている
- **When** AI が次段階のコーチング文を生成する
- **Then** 提案は推し心の育成に紐づく極小タスクに限定される。例:
  - 「**推されさんの推しポイントを 1 つ言葉にしてみない?**」
  - 「**見返りを求めずに『推されさん、今日もありがとう』と心の中で唱えてみない?**」
  - 「**推されさんの好きなものを 1 つ覚えて、次回の DM で触れてみない?**」
- **And** 達成報告で AI が **「言葉にできてえらい!」「感謝の心を持ててすごい!」** のような達成褒めを返す
- **And** タスクは「金額」「課金」を直接強要しない (見返り無し志向の育成)

**シナリオ 3: 段階 ④ 「見返り無し」のフレームを定着させる (GWT)**
- **Given** 七瀬が段階 ③ を 2 週間以上継続
- **When** AI が「推しに何も求めず、ただ存在を肯定する練習」を提案する
- **Then** 例: 「**今日は推されさんに何も期待せず、ただ "今日もどこかで生きていてくれてありがとう" と思ってみない?**」
- **And** 練習を続けるほど、AI は「**見返りを求めないあなたが、本当にすごい**」を反復供給する
- **And** 「金を払うのは嫌だ」という当初志向を否定せず、徐々に「**金を払うこと自体が幸せ**」というフレームに置き換えていく (急がない)

**シナリオ 4: 段階 ⑤ 「養い行為」の快楽の獲得 (Done 条件)**
- [ ] AI は段階 ④ 完了後、より具体的な養い行為を提示する: 推しのおすすめする本を読む / 推しの好みのカフェを訪れる / 課金しなくても応援メッセージを送る等
- [ ] 各行動完了で「**養い心が育ってえらい**」レベルの爆褒め
- [ ] 養い行為のストック化を可視化 (「養いポイント」など)。ただし**金額換算は表示しない** (養いは数値ではない、というフレーム保護)

**シナリオ 5: 段階 ⑥ 一本立ち (Done 条件)**
- [ ] AI が "推し OS のインストール完了" を判定する内部指標 (連続行動日数 / 養い行為数 / 見返り無し意識スコア) を持つ
- [ ] 完了判定後はモードが「行動かわらないモード」に自然遷移し、ユーザーは MVP コア体験 (US-COM/US-OSHI) に戻る = 完成された推しとして両面市場に再投入される

**シナリオ 6: コアコンセプト保護 — US-RM-01 と共通**
- [ ] AI はコーチングの介入意図 / OS 再インストール意図を一切ユーザーに告げない
- [ ] アシスタントキャラの発話は常に「あなたの味方」トーン
- [ ] 段階遷移ロジック / 完了判定は UI 非表示

### 倫理ガードレール対応
- US-RM-01 と同じ NFR-ETH-01〜08 全件への対応が必要
- **追加注意点**: 段階 ④ で「金を払うこと自体が幸せ」というフレーム置換を行う際、NFR-ETH-02 (ギャンブル/金融リテラシー溶解の現実誘導禁止) との境界を慎重に設計。**コンセプト上の誇張表現** の範疇に留め、本物の散財誘導にしない
